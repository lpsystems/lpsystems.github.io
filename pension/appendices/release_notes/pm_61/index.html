Release Notes 6.1

Important Notes In Relation To The Euro

  1. You will need to amend Scheme User Defined Screen, User Defined Field and User Defiled Array field formats from N2 to MO (see here for further details on how to do this) if you wish these amounts to be converted by any of the new Pension Master Euro functionality. All user defined field types of type Nx (where x = 1,...,6) will not be converted to Euro's as they are assumed to be non-numeric values.
  2. All quotation results will be assumed to be monetary amounts for the purposes of any Euro conversions and will thus be converted to Euro's as and when the scheme base currency is changed to ther Euro. If the results of any quotations are not monetary amounts you will need to re-run the quotations in order to obtain valid non-monetary amounts.
  3. Any monetary amounts referred to in either Scheme Notes or Member Notes will not be converted to Euro's as there is no way for Pension Master to distinguish between monetary amounts and other numeric items in the notes. All users of Pension Master should be made aware of this.

Formula File Maintenance

Extended the avg() and avgmax() functions in the formula file to allow a fifth averaging basis of dynamisation. The relevant help text is as follows:

avg(field,n,fld1,fld2)

OVERVIEW

Use the avg function to calculate the average of a history field over a particular number of years.

SYNTAX

avg(field,n,fld1,fld2)

EXPLANATION

field - is a field on the salary history or contribution history files (pe_defsal or pe_defhst_b respectively).

n - is the number of years to average the history field over.

fld1 - is the effective start conditions of the averaging.

fld2 - is the basis of the averaging.

NOTES

  1. If n = 0, the function will return a value of 0.
  2. n can take any value less than 40. i.e you can average the history field over no more than 40 years.
  3. fld1 can take the alphabetic values 'e' and 'l'.
    • 'e' - will commence the averaging period from Calculation Date.
    • 'l' - will commence the averaging period from Calculation Date but will ignore the latest entry.
  4. fld2 can take the values 1, 2, 3, 4 and 5.
    1. will average extracted amounts using the highest amount if there is more than one in the same calendar year.
    2. will average extracted amounts using the first amount if there is more than one in the same calendar year.
    3. will average extracted amounts recorded at the renewal dd/mm as recorded on Scheme File Maintenance. A warning will be generated if there are missing years in the salary history.
    4. will average extracted amounts using the period the amount was effective for.
    5. will average extracted amounts recorded at the renewal dd/mm as recorded on Scheme File Maintenance and dynamise using the appropriate RPI values. N.B. The correct calculation of an average salary using this basis is dependant on the relevant RPI values being present in MLI/RPI File Maintenance. A warning will be generated if there are missing years in the salary history.

EXAMPLES

avg(pds_pensal,3,e,1)
avg(pds_grsal,1,l,4)
avg(pdb_ee_ord_c,4,e,2)

avgmax(fld,n1,n2,fld1,fld2)

OVERVIEW

Use the avgmax function to calculate the best band of consecutive years history over a given period.
e.g. to calculate the best three year average gross salary over the last ten years.

SYNTAX

avgmax(field,n1,n2,fld1,fld2)

EXPLANATION

field - is a field on the salary history or contribution history files (pe_defsal or pe_defhst_b respectively).

n1 - is the number of years to find the best consecutive n2 years average over.

n2 - is the number of consecutive years to average the history field over.

fld1 - is the effective start conditions of the averaging.

fld2 - is the basis of the averaging.

NOTES

  1. If n1 = 0 or n2 = 0, the function will return a value of 0.
  2. n1 can take any value less than 40. i.e you can average the history field over no more than 40 years.
  3. If n2 > number of years history for a particular member then the function performs the average over the number of years history data available. e.g. if you try to use avgmax(pds_grsal,10,3,e,1) and there is only 2 years gross salary details for a particular member, the function will look at consecutive two year averages of gross salary to arrive at the result.
  4. fld1 can take the alphabetic values 'e' and 'l'.
    • 'e' - will commence the averaging period from Calculation Date.
    • 'l' - will commence the averaging period from Calculation Date but will ignore the latest entry.
  5. fld2 can take the values 1, 2, 3, 4 and 5.
    1. will average extracted amounts using the highest amount if there is more than one in the same calendar year.
    2. will average extracted amounts using the first amount if there is more than one in the same calendar year.
    3. will average extracted amounts recorded at the renewal dd/mm as recorded on Scheme File Maintenance. A warning will be generated if there are missing years in the salary history.
    4. will average extracted amounts using the period the amount was effective for.
    5. will average extracted amounts recorded at the renewal dd/mm as recorded on Scheme File Maintenance and dynamise using the appropriate RPI values. N.B. The correct calculation of an average salary using this basis is dependant on the relevant RPI values being present in MLI/RPI File Maintenance. A warning will be generated if there are missing years in the salary history.

EXAMPLES

avgmax(pds_pensal,10,3,e,1)
avgmax(pds_grsal,3,1,l,4)
avgmax(pdb_ee_ord_c,4,1,e,2)

Scheme User Defined Screen

Several new field formats have been added to this screen.

'MO' - Monetary Amount
'Nn' - Number with n decimal places (n = 0-6).

'MO' has been added in the advent of Euro conversions which will allow monetary values held on Pension
Master to be converted appropriately.
'Nn' has been added to extend the flexibility of the system and allow users to record amounts with up to
six decimal places.

System Administrators: You will need to ensure that administrators are aware that they will need to amend Scheme User Defined Screen field formats from N2 to MO if they wish these amounts to be converted by the 'Toggle Euro' function.

The relevant amended section of the User Manual reads as follows:

FIELD DESCRIPTIONS/FORMATS - Enter the field descriptions and field formats for the schemes Scheme User Defined Details. You can leave blank lines to space your record out and you must enter a description if you enter a format.

The list of valid formats can be selected by clicking on the 'Search' button. These are listed below:

  • DT - Date - dd/mm/yyyy
  • IN - Indicator - Single character (uppercase only)
  • TX - Text - Up to ten characters of free text
  • MO - Monetary Amount - Number with two decimal places
  • Nn - Number with n decimal places. n can equal 0, 1, 2, 3, 4, 5 or 6 e.g. N3 - Number with 3 decimal places.

  • (N.B. Due to field formating, a field with a format of Nn cannot contain more than 9 numbers e.g. a field with a format of N4 can only have up to 5 numbers before the decimal point and 4 after, similarly and field with a format of N1 can only have up to 8 numbers before the decimal point and 1 after).

FIELD VALUES

- Enter the field values for the schemes Scheme User Defined Details. You will only be able to enter a value if the field description and format are present. This will be checked by the system to ensure it is a valid entry for the entered field format. If a value is updated for a field with an associated format of MO or Nn, the existing value will be redisplayed left justified.

Existing field formats of NI (Numeric Integer) and ND (Numeric Decimal) will be automatically converted to N0 and N2 respectively.

Transfer In

The members Transfer In screen has been changed from a one screen format to three screens.

Upon entry to the screen any existing transfers for the current member are displayed in a window. A specific transfer can then be selected and the transfer details displayed on the screen. The menu has been changed slightly so that the second and third screens of details can be seen by pressing the "2" or "3" keys or by moving the highlight bar with the arrow keys and pressing Return. In a similar way the first screen of details can be seen by pressing the "1" key or by using the arrow keys.

The Add option is only available from the first screen. This allows the user to enter the first screens data then automatically switch to the second and then third screens. Upon clicking on 'Ok' on the second screen the data from both the screens is saved to the file. Also if you click on 'Cancel' on the second screen data that was previously entered on the first screen will not be saved.

Browsing of a members Transfer In details can be performed on either screen, the same also applies to any Updating required.

Code Maintenance

The Additional Benefit Code option on the System Data menu has been replaced by the Codes Maintenance option. This new option allows five different types of Code to be entered through the same screen as follows:

  • A - Additional Benefit Codes
  • D - Deferred Pensioner Codes
  • I - Transfer In Codes
  • O - Transfer Out Codes
  • P - Pensioner Codes

These codes are then used when setting up the relevant record on the member file. The Deferred Pensioner and Pensioner Codes will be used by new screens in the next release of Pension Master.

Transfer In/Out Data

The file old pe_transfer database table (which used to hold details of both transfer in's and transfer out's) has been replaced by new separate tables, pe_tvin and pe_tvout. The installation procedure creates the new files and copies the existing data. Because the new Transfer In/Out screens rely on individual codes (as set up in Codes Maintenance) to access the additional data, the program will automatically create several new codes as follows:

Transfer In's TPxxxxxx with a description of "Transfer Pension"
Transfer Out's TCxxxxxx with a description of "Transfer to a New Company"

The number section of the code will depend entirely on the number of Transfers for that member. e.g. if the member already has 2 Transfer In's, then the codes will be named TP000001 and TP000002 respectively.

Database Maintenance

Added a new module, Database Maintenance to Pension Master. This is available under the GUI version of Pension Master only and allows you to set up and maintain extra databases in addition to the default 'pension' database.

This can be useful when, for example, setting up and testing formulae, where you do not wish the formulae to be available on a live system until they have been properly de-bugged. See the Database Maintenance section of the User Manual for further details.

If your Pension Master has multiple databases, you can tell which one you are connected to by looking at the titlebar of the Pension Master window - it will have a (dbasename.dbs) at the end of it, where dbasename is the name of the database you are using.

System Administrators: You will need to make this extra menu option available to users as appropriate by granting access using the User File Maintenance module of Pension Master. By default all new menus and menu options added to Pension Master will not be switched on for any users.

Logon Screen

This is available for the GUI version of Pension Master only and is to be used as the first screen that users see when clicking on the Pension Master icon from their desktops. This allows the user to log into Pension Master by use of the standard username and password sequence. It also allows users to change their password as and when required.

All passwords are stored in a one-way encryption format making it extremely difficult to crack passwords and additional controls over the choice of passwords have been introduced in the System Parameter Maintenance module of Pension Master.

It is not now possible to run individual Pension Master modules in the GUI environment without first logging into Pension Master using this login screen, thus further strengthening the security that is built into Pension Master.

The way Pension Master logs database changes is now set via a second screen in System Parameter Maintenance as detailed above.

Use the date_ymd function to calculate a date, given the values for the date's day, month and year e.g. date_ymd(1998,10,12) will evaluate to the date 12/10/1998.

SYNTAX

date_ymd(year,month,day)

EXPLANATION

  • year - year for calculation
  • month - month for calculation
  • day - day for calculation

NOTES


  1. year, month and day must be either a constant value entered in the appropriate format (YYYY for the year, MM for the month or DD for the day) or a reference to the result of another line of formula code within the current formula e.g. @1.
  2. if the result of the function call does not return a valid date e.g. if the function arguments are results of previous lines of your formula which do not evaluate to correct years, months or day values, a value of 0 will be returned and a message will be written to the Quotation Warnings file informing you of this. You should then examine your formula as it is quite likely that the result of the formula will be incorrect.

EXAMPLES

date_ymd(1998,@1,10)
date_ymd(@1,@2,@3)

Added new GMP functions as follows:

pre_88_gmp_death(p,a)

OVERVIEW

Use the pre_88_gmp_death function to calculate the Guaranteed Minimum Pension (GMP) accrued prior to 6 April 1988, revalued to date of death.

SYNTAX

pre_88_gmp_death(p,a)

EXPLANATION

  • a - is a single character field that takes the value 'Y' or 'N'. It denotes whether to include any transfered-in GMP in the GMP calculations
  • p - is a single character field that denotes the type of GMP revaluation as follows:
    • L - Limited rate revaluation
    • F - Fixed rate revaluation
    • S - Section 148 orders. These are assumed to be the same as for Fixed Rate revaluation.

NOTES

  1. Date of death is taken from the Date Of Death field on the Death Details screen in the member record. If this is blank, date of death is assumed to be the Date Left field on the Basic Member Details screen. If this is blank, then a warning will be written to the Quotation Warnings Report and this function will return a value of 0.
  2. If date of death is later than State Pension Date then the GMP is revalued up to State Pension Date using Fixed, Limited or Section 148 orders, and between State Pension Date and Normal Retirement Date at 1/7% per week, providing the period between State Pension Date and Normal Pension Date is at least seven weeks.

EXAMPLES

pre_88_gmp_death(L,Y)
pre_88_gmp_death(F,N)
pre_88_gmp_death(S,Y)

Scheme Level Code Allocation

The Codes Maintenance option has been extended to allow codes to be allocated to specific Schemes, rather than having system-wide global codes.

An additional screen has been introduced to the Scheme File to facilitate this. This screen is used to change the codes that have been allocated to the Scheme. When a members additional benefits, transfer in, transfer out, deferred pensioner or pensioner details are updated a lookup list for the code to use for the particular record will only show the codes relating to the members Scheme.

Member File Maintenance

The Delete option in the Member File has been modified so that instead of deleting the whole member record regardless of what screen in the member file you are in, the delete option is more context sensitive. What is deleted by using the Delete option is dependent on the screen you are in as follows:

  • Basic Member Details screen - the Delete option deletes all data for the current member, including all related member data e.g. salaries and contributions. The whole of the member record will be deleted from your Pension Master system.
  • Additional Benefits screen - the Delete option deletes the currently selected additional benefits record in for the current member.
  • Transfer In screen - the Delete option deletes the currently selected transfer in record for the current member.
  • Transfer Out screen - the Delete option deletes the currently selected transfer out record for the current member.
  • Refund Of Contributions screen - the Delete option deletes the refund of contributions details for the current member.
  • Deferred Pensioner screen - the Delete option deletes the deferred pensioner details for the current member.
  • Pensioner screen - the Delete option deletes the pensioner details for the current member.
  • Death details screen - the Delete option deletes the death details for the current member.
  • User Defined Screens - the Delete option deletes the currently selected user defined screen values for the current member.
  • User Defined Arrays - the Delete option deletes the currently selected user defined array values for the current member.

The Delete option is only available in the above screens and will not be displayed on the menu if you are in any other screen as it is not relevant.

Further details can be found in the section Deleting Member Records elsewhere in this User Manual.
