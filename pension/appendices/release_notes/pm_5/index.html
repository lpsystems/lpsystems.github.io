Release Notes 5.02b6

Introduction

This is a major new release of Pension Master packed with new features that both extend the capabilities of the software and make it easier to use. The main changes can be summarised as follows:

  • Introduced a Quotation Type feature so as you can group together logical sets of benefit calculation formulae.
  • Re-designed Formula File Maintenance so as to allow an unlimited number of lines of formula code per formula. The F1 and F2 keys are now available for inserting/deleting lines of formulae, with appropriate re-numbering of subsequent formula lines.
  • Replaced the old Age/Sex Factor Table and Term Related Factor Table modules with a new improved Factor Table Maintenance program.
  • Integrated Workflow Management facilities are now built into Pension Master.
  • Added a new Renewal Data Request report allowing you to print off blank end of year renewal returns for sending out to clients.
  • Formulae can now be directly accessed from the Member Report, Full Members Listing and Mail-Merge Integration Report rather than having to run a Batch Quotation Run prior to running your report.
  • You can now access more database tables using the Member Report, Full Members Listing and Mail-Merge Integration Report. This will allow you, for example, to print out units from more than one investment fund on the same report.
  • Introduced new Letter/Cert option in Company File Maintenance for sending out mail-merges to companies within a scheme.
  • Added new Salary Listing Report to allow you to print off salary details for a selected group of members for a specified date.
  • Added new Investment Value Summary Run to allow you to update a batch of members Investment Value Summary screens in Member File Maintenance and to print off a report showing up to date investment values for a selected group of members, based on either a unit bid or offer price.
  • You can now automatically generate a list of relevant members when adding a batch in the Batch Data Entry module. This will save you time as you will not have to input the member number on each line of data you input.

Details Of The Changes

  1. Added F8 lookup facility to Unit Posting program on Scheme, Company, Category and Status fields.
  2. Removed old pe_num, pe_bnum and pe_nxtsch database tables and modified all programs accordingly. These tables used to hold the values for the next member number, batch number and scheme number respectively. The programs now default these fields to 0 when adding a member, batch or scheme respectively and when Esc is pressed to save the details, the relevant member, batch or scheme number is displayed to the screen and used to save the record accordingly.

  3. This removes the old problem of, say, when adding a new member, the next member number is displayed to the screen, but if you subsequently press Del to abort the input of the new member, the next time you go to add a member, the last member number has been skipped over. i.e. we are now ensuring a strict sequence of consecutive member numbers.
  4. Changed List report in Scheme File Maintenance so that each new scheme is not started on a new page i.e. schemes are now listed consecutively, one after the other.

  5. Added facility to call formulae instead of fieldnames when processing a Member Report, Full Members Listing Report or a Mail-Merge Integration Report. This is done by entering the following syntax:

    #calc(formula_code)

    where formula_code is a 3 character formula code for the current scheme. The formula code will be syntax checked on entry and if it does not exist for the scheme implied by the current selection criteria, an error message will be displayed and you will have to re-enter a valid formula code.
    The formula corresponding to the formula code you entered will be dynamically calculated and output to the report as the report is running, thus removing the necessity to remember to run some quotes before using either of these three reports.

    This will not affect the quotation values that are displayed on the Quotation Screens in Member File Maintenance.

  6. Member Report, Full Members Listing Report and Mail-Merge Integration Report now show the current member number being processed to the top of the screen so as to give you some idea of the progress of the report.
  7. Added an F8 lookup for Formula Code to Member Report, Full Members Listing Report and Mail-Merge Integration Report. In the report fields screen, the default F8 lookup is for fieldname, but if you have any field beginning with the # symbol, the lookup changes to that of a valid formula code for the scheme implied by the current selection criteria and once the formula code is selected from the lookup, the line is written back to the screen field in the #calc(formula_code) format.
  8. Added an F8 lookup for Scheme Code, Company Code, Category Code and Status Code to Member Report, Full Members Listing Report and Mail-Merge Integration Report.
  9. Extended F9 context-sensitive help displays in the Member Report, Full Members Listing Report and Mail-Merge Integration Report.
  10. Added an F8 lookup for the six Group By fields in the Member Report, Full Members Listing Report and Mail-Merge Integration Report. This lookup is restricted to the Member File (pe_mem) as the user is only permitted to sort records by fields on this table in these reports.
  11. Added facility to pass arguments when using certain fields in the Member Report, Full Members Listing Report and Mail-Merge Integration Report. This allows you, for example to select unit totals from a particular investment fund, which was previously not possible using Pension Master. Details of the arguments required are shown in the following table:
    Table Description Argument 1 Argument 2
    pe_defhst_b Contribution History Body pdb_inv_fund -
    pe_defhst_h Contribution History Totals pdh_inv_fund -
    pe_dep Dependants Details pd_num -
    pe_mifund Member Investment Choice pmif_ifund -
    pe_mquote Member Quotation Results pmq_type pmq_index
    pe_not Member Notes pn_num -
    pe_transfer Transfer In/Out Details ptv_type ptv_num

    As an example, say you wanted to print a benefit statement with the first five nominated beneficiaries for each member (by reference to the pe_dep database table), you would enter:

    pd_dep(1)

    to select the first dependants name, and just add four extra fields to the report to select each of the other dependants in turn.
    The syntax of the arguments is checked by the report as you add the fieldname and you will be warned of any errors or inconsistencies if and when they occur.

  12. Replaced the old Age Related Factors and Term Related Factors programs by a new improved Factor File Maintenance program. This copes with both of the old factor table types and allows more efficient use of the indices for each factor. Previously, if you have a table of factors, say from age 20 to 60, and decided that you really needed factors down to age 18, the extra ages would have to be added to the end of the table, after the factor for age 60. This lead to some strange screen displays as the factors could not be displayed in age order.
  13. This is now fixed and we have introduced two new fields, Range From and Range To to the factor table header. These are to be used to adjust the range of ages (or terms) for your factors as you are no longer permitted to actually edit the ages (or terms) directly.
    Simply changing either of these fields for an existing factor table will cause the factors to be adjusted so as they only related to all ages (or terms) between the new values for Range From and Range To. If this results in new factors being added, they are defaulted to zero and you will be required to fill in their values for each new index.
    If any of the factors are zero when you try to save your factor table, you will be warned of this and asked if you want to continue and save the table or input some non-zero factors.
    Also the old limit on 54 individual factors per factor table has been removed. You can now have an unlimited number of factors per factor table as required.

    The System Administrator will need to use the Menu Maintenance program to manually add this option to the Pension Master File Maintenance menu and then use User File Maintenance to grant permissions to all suitable users so that they can use this new module.

  14. Formula File Maintenance has been re-written so as to remove the limit of 100 lines per formula. You can now have an unlimited number of lines per formula. We also added support for the F1 and F2 keys to insert and delete lines in the formulae respectively. When these keys are used, all subsequent lines of the formula referring to other lines (in the @line_number notation) are renumbered accordingly.
  15. Formula File Maintenance now has one screen which scrolls round to allow you to edit the formula lines. This is in place of the five separate screens in the old Formula File Maintenance.

    #pmq_current(NORM,1)

    If the quote is not found, the function call returns a value of zero and prints an appropriate error message to the quotation error log.

The data is sorted by company and then by surname and forename, with each company starting on a fresh page. Details pre printed on the report are:
    • Scheme code and name
    • Company code and name
    • Member number
    • Member name
    • Date of birth
    • Sex
    • Marital status

    The report automatically prints boxes around each line of member details so as to aid the employer in filling in the schedule by hand.
    The administrator can then define titles for up to 5 items of data that they are requesting from the employer which are in turn printed on the report. The renewal date used for the report is automatically determined by reference to the renewal day and month on Scheme File Maintenance and the current year.

    The System Administrator will need to use the Menu Maintenance program to manually add this option to the Pension Master Report menu and then use User File Maintenance to grant permissions to all suitable users so that they can use this new report.

  48. Added a new function write(field,@line_num) to Formula File Maintenance as follows:

    write(field,@line_num)

    OVERVIEW

    Use this to write back the result of a formula to either the deferred pensioners file (pe_dpen) or the pensioners file (pe_pen). This could be useful when doing some final calculations for a member and you wish to formally record the results of the calculations in either the deferred pensioner or pensioner screens in Member File Maintenance.

    SYNTAX

    write(field,@line_num)

    EXPLANATION

    field - a valid numeric field on either the deferred pensioner table (pe_dpen) or the pensioner table (pe_pen). This denotes the field that you wish to write the result of this formula to.

    @line_num - the line number of the line you wish to output the result from e.g. if you wish to write the result of line 5 to the field pp_precom_pen in the pensioner table, enter: write(pp_precom_pen,@5)

    NOTES

    1. If there already exists a value in the field specified for the member, it will be automatically overwritten with the result of the formula.
    2. You can only reference line numbers before the current formula line i.e. if your write() function is on line 6 of your formula, you cannot reference lines >= line 6 in your write() function call.
      3. For example: write(pp_precom_pen,@10) or write(pdp_grpen_nrd,@2)
  49. Removed the link between Category Code and Company Code so that Categories are now only sepefic to particular Schemes and not particular Companies within particular Schemes. This has greatly reduced the number of Categories necessary on Pension Master which in turn will simplify the adding of things like new Formula Codes to Categories.

  50. Calls to the goto function in Formula File Maintenance must now be in the format:

    goto @line_number
