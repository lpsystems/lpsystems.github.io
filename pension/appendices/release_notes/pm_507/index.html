Release Notes 5.0.7

Scheme File Maintenance

Basic Scheme Details

This is essentially the same Basic Scheme Details screen as is currently in Pension Master. Additional fields have been added as follows:

  • Alternative scheme code - to be used for the an alternative scheme code (if applicable). This will be a ten character alphanumeric field which will also be added to the list of possible search options in Member File Maintenance. This field will default to the same value as the 'Scheme Code' if not filled in otherwise.
  • Revenue Reference Number (Ireland). For UK schemes this will be labelled 'ECON'.
  • Pensions Board Reference Number (Ireland). For UK schemes this will be labelled 'SCON'.
  • (UK only) - PSO Reference Number. For Irish schemes, this field will not be displayed. This will be displayed after the 'Pensions Board Reference Number' shown below.

All fields would be held on the scheme file and so would be available to Member Reports and for mail-merge to Microsoft Word.

The three old State Pension fields will be dropped. The relevant data is now held in the 'Statutory Amounts Maintenance' program and is accessible via the formulae using the new stat() function call in release 5.0.5.

The Contracted Out Basis field has also been dropped. This will allow access to the NI History screen for Defined Contribution schemes.

Investment Funds

This is essentially the same screen as the old 'Documentation/Investment Funds' screen except that the documentation section of the screen has been separated out into a new 'Actuarial Details' screen - see later for further details.

This screen has been left deliberately sparse to make space for future planned developments to Pension Master with regard to extending the capability of investment funds.

Renewal Details

This is a new screen that holds, for each constituent company within a scheme, details of the progress of the renewal for the current scheme year. An option will also be available so that the user can view/update details from previous years renewals for the current scheme.

This allows the administrator to complete the renewal process as and when data is received, in a case where each company runs their own individual payroll and hence sends in their own data. This avoids having to wait for the whole scheme to send in data before any work on the renewal can be processed.

The 'Full/Partial Data Received' field takes the following values:

  • P - partial data received.
  • F - full data received.
  • N or ' ' (blank) - no data received yet.

There will be a separate 'Renewal Status Report' to analyse the renewal data across a particular scheme or all schemes.

Actuarial Details

This is a new screen.

The first part of the screen is essentially the same as the Documentation part of the old 'Documentation/Investment Funds' screen. Additional fields have been added as follows:

  • Date Last Actuarial Valuation Accepted
  • Date Last Member Report Accepted
  • Non-selection limit
  • Reassurance threshold limit
  • PHI Cost
  • Current Funding Rate

All fields would be held on the scheme file and so would be available to Member Reports and for mail-merge to Microsoft Word.

Address Details

This is a new screen.

This screen would hold details of the addresses in relation to the scheme. Separate screens with the same format shown as below would be implemented for:

  • Scheme Address
  • Trustees address (if only 1 main trustee address)
  • Administrators address
  • Actuary's address
  • Auditors address
  • Legal advisors address
  • Miscellaneous address 1
  • Miscellaneous address 2

All fields would be held on the scheme file and so would be available to Member Reports and for mail-merge to Microsoft Word.

Scheme Level User Defined Fields

This is a new screen.

This screen is designed to hold up to twenty scheme-wide user defined fields in the same kind of manner as the existing member-level user defined screens.

For each scheme-level user defined field required, the user would enter a description, a format for the field and the actual field value.

The field format's can any of:

  • DT - date
  • IN - indicator (Y/N etc.)
  • NI - numeric integer
  • ND - numeric decimal
  • TX - free text

Two of these fields would be used to hold the Life 400 pension and life policy numbers.

Member Screen Selection

This is a new screen that allows the System Administrator to customise which screens are available in the member record on a scheme by scheme basis. Also it introduces the concept of screen based security levels in the member record, so as, for example salary details can only be viewed by a suitably senior administrator.

The fields are as follows:

DESCRIPTION - This is a view only field that records the internal Pension Master description e.g. 'Basic Member Details' for each member screen in turn.

DISPLAY - Display this screen for members of this scheme. Valid values are 'Y' or 'N'.

SECURITY LEVEL - Enter the security level for this screen. Valid values are 1 to 99, with 1 being the highest security level.

Broker Details

This is a new screen.

This screen will hold details of the broker assigned to the current scheme by cross reference to the new 'Broker File Maintenance' module (see later in this document for further details).

The user enters the 'Broker Code' field, which has a corresponding pop-up F8 picklist key and the broker details are displayed to the screen and available in the Scheme Report.

Category File Maintenance

Basic Category Details

This is essentially the same as the basic category details screen as is currently in Pension Master. Additional fields have been added as follows:

  • CONT % - EE ORD - to be used for the current employee contribution rate for the scheme. This applies only for defined benefits schemes at present. Functionality for defined contribution schemes in this regard will be introduced in due course.
  • DIS BENEFIT - Min Age/Service For Membership - to be used when adding a member who is included in the scheme for life assurance only, and will be used to validate the new 'Date Joined For LA' field on the 'Basic Member Details' screen in 'Member File Maintenance'.
  • Mid Year Entrants Allowed (Y/N) - self explanatory. There are two fields against this heading, one for Full Members and another for DIS Members. These will be used in the validation of new members for the current scheme and category combination. These fields will default to 'N' for all categories. This will also be used by the for pro-rating within the appropriate reports.

Benefit Definitions

This is the same as the old 'Benefit Definitions' screen except two extra fields have been added as follows:

  • Original Status Type - this will allow Pension Master to only display Quotation Types that are relevant to the current member's status i.e. only allow the user to run quotes that are relevant for the current user. This field would match against the members current status type e.g. A - Actives. Only members that currently have this status type would have this Quotation Type displayed in the list of available options. A code of 'G' would be created as a global code which would allow the Quotation Type to be available to members of all status types.
  • Finalised Status Type - this will allow Pension Master to update the members status code on finalising Quotation results. This would work in the same way as the new 'Original Status Type' field outlined above except that when the user tried to finalise the Quotation results, they would be presented with a pop-up list of applicable status codes in the specified status type group. The member's status history screen would also be updated at this time to reflect the change in status code. This code can also be set up to prevent users from being able to finalise a quotation, by using a code of 'X', and to only allow the write back of quotation results without updating the members status, date left or the quotation finalised flag, by using a code of 'R'.

Status File Maintenance

This is the same as the current Status File Maintenance screen except that an extra field has been added as follows:

  • Life Assurance Only (Y/N)

This field will default to 'N' for all status codes, but can be set to 'Y' for those status codes that are for life assurance only type benefits.

This will then be used within the Pensions Board Fee Invoice Report to determine which active members are pension members (and not just life assurance only members) and hence should be included in the Pensions Board Fee Invoice calculations.

It is also used on the Underwriting/Life Assurance Report to determine which members to total up salary roll for when working out the total funding amount and the total employee contributions due.

Member File Maintenance

Basic Member Details

This is the same basic member details screen that is currently in Pension Master with the following exceptions:

  • Changed 'Date Entered Scheme' field label to 'Date Joined For Pension' to be consistent with the new 'Date Joined For LA' field.
  • Changed 'Date Entered Category' field label to 'Date Joined For LA'. This is to be used to record the date the member was covered for life assurance by the scheme from.

Validation of the 'Date Joined For LA' field would be against the entry conditions headed 'DIS BENEFITS' on the basic category details screen in Category File Maintenance (see elsewhere in this note for further details).

With regard to mid year entrants, Pension Master will check to see whether the 'Mid Year Entrants Allowed (Y/N)' flags on the Category File are set to 'Y' or 'N' for normal and DIS benefits as appropriate.

If they are set to 'Y' then Pension Master will function exactly as it does as present.

If set to 'N' then Pension Master will only allow the following dates to be dates that fall on the same day and month as the Renewal Date in the appropriate category for the member (see the Basic Category Details screen in Category File Maintenance for further details):

  • Date Joined For Pension
  • Date Joined For LA
  • Qualifying Service Date

In the case of members with entry criteria specified on the category file, who are also not allowed to join the scheme as mid-year entrants, the appropriate fields on the member file will default as normal, but will be to the renewal date immediately after the date they would become eligible to join the scheme, had mid-year entrants to the scheme been allowed.

When adding a new member the effective date for Company, Category and Status History entries will default to Date Joined For LA (pm_dect). If this date is not present then the system will use Date Joined For Pension (pm_des). If this date is also blank then the system will prompt the user to enter an effective date when they attempt to save the Basic Details screen.

The security rating checks have also been slightly amended. Users who do not have a high enough security rating to access member data for a particular scheme, member or screen will have an appropriate error message displayed. The system will then display the selected member record with only the basic member details shown in the screen header box. This allows users to know that a member is present on the database but they do not sufficient access rights to view or update the members details.

Member Indicators

This is the same member indicators screen as is currently in Pension Master with the following exception:

  • New 'Member Medically Loaded' field

This field takes the values 'Y' or 'N' and will automatically default to 'N'. It is used to indicate to the administrator whether this member has a medical loading imposed. It is purely an indicator and no supporting calculations are deemed to be necessary. The user will detail the loading and the benefit to which it applies on the notes screen.

The medically loaded members life assurance premium will be calculated normally, i.e. without applying the loading.

The value of this field is printed on the new 'Underwriting/Life Assurance Report' for use by underwriters.

Member Quotation Screen

This has been changed to only display appropriate Quotation Types depending on the members Status Type, the Quotations Original Status Type and finalised status.

The old Termination Date (pm_dterm_dt) field seen in previous releases of Pension Master has been replaced with a new field - Calculation Date. This will serve as the effective date of the Quotation i.e. Date of Leaving, Date of Retirement etc. Termination Date will be calculated in the background as before (defaulting to Calculation Date or Date Left if present) so existing occurrences of pm_dterm_dt in formulae will continue to work as before. The creation of this Calculation Date field is held for each Quotation Type as opposed to Termination Date which is a single field that held the date entered for the last Quotation run for the member. The Calculation Date can be used in formulae by using the code pmqh_calcdate(TYPE) were TYPE is the appropriate Quotation Type.

Once a Quotation Type has been selected, the user will be able to run the calculation as before.

Once a Quotation has been run, the user can opt to finalise it providing that the Finalised Status Type for the Quotation Type on the Category File allows it. The action of finalising a Quotation has been extended to update the members Status History and Date Left if the member is changing from an active status to a non-active one. This will only happen if the Finalised Status Type has not been entered as 'X' (Prevent Finalise) or 'R' (Write back results only). The act of unfinalising a Quotation will remove the entry from Status History and reset the members Date Left to blank if they are changing from a non-active status to an active one.

Member Search Facility

Include the new 'Alternative Scheme Code' field on the 'Basic Scheme Details' screen in 'Scheme File Maintenance' in the list of possible search criteria for members of a pension scheme.

Broker File Maintenance

This is a new module which would hold the following details for each possible broker:

  • Broker Code
  • Broker Name

  • Broker telephone number
  • Broker contact name

This data would then be linked in to individual scheme records via the new 'Broker Details' screen in 'Scheme File Maintenance'.

The 'Broker Code' field is a 5 character field that permits all alphanumeric characters.

Underwriting/Life Assurance Report

This is a new report and will first of all calculate the non-selection (or free cover) limit. This will be done by looking at all members of the scheme whose status type is 'A' (i.e. all active members of the scheme), multiplying the result by 10,000 and subjecting this to an overall maximum of 700,000. The resulting value will automatically update the new 'Non-Selection Limit' field on the new 'Actuarial Details' screen in 'Scheme File Maintenance'.

A report will then be produced by analysing the life assurance benefit and the capitalised spouses in service benefit values that have been held on an appropriate user defined screen. This will allow the user to select by scheme code and for all the members within that scheme, output a report as follows:

This report would also show the amount for joiners in the last year, pro-rated by reference to the 'Date Joined For LA' (date the member joined the scheme for death benefits) field and up to and including the day before the renewal date.

All active members (those members with a status type of 'A' corresponding to their respective status code) will be used in both the calculation of the NSL and the corresponding report.

Pensions Board Fee Invoice Report

This is a new report and is applicable to Irish based schemes only.

It is required to calculate and print out the amount of the fees due for the scheme in respect of the annual Pensions Board Fee. This will then be put into an invoice and can used to bill the client for the amount due. The calculation is based on the number of active pension scheme members, excluding members who life assurance only.

All active members (those members with a status type of 'A' corresponding to their respective status code), but excluding life assurance only members will be used in both the calculation of the amount due. This will be by reference to the new 'Life Assurance Only' field on 'Status File Maintenance' (see elsewhere in this document for further details).

Premium Calculation Report

This is a new report.

It is required to show the amount of premium due in respect of each scheme.

The user selects the scheme for which they wish to run the report and indicates which type of salary to use for each of the employee and employer contributions e.g. A = Gross Salary, B = Scheme Salary etc. etc.

The data used is as follows:

  • Employee Contributions is calculated by using the 'EE ORD CONT%' and the relevant salary details.
  • The total salary roll is calculated by summing up all the latest salaries for active members of the scheme.
  • Funding Rate is taken from the new 'Actuarial Details' screen in 'Scheme File Maintenance'
  • Life assurance cost will be obtained in the same way as the data for the Underwriting/Life Assurance Report mentioned earlier.
  • PHI Cost is taken from the new 'Actuarial Details' screen in 'Scheme File Maintenance'. This field is manually updated by the user from an external PHI system.
  • The Medical Loading field is a run time input by the user on the report and can be used for any additional items that are required to be offset in the calculation of employers contribution.

This report would also show the amount for joiners in the last year, pro-rated by reference to the 'Date Joined For LA' (date the member joined the scheme for life assurance) field and up to and including the day before the renewal date.

Renewal Status Report

This is a new report.

It is required to analyse the details held on the new 'Renewal Details' screen in 'Scheme File Maintenance' in order to allow the administrator to analyse the current status of renewals on Pension Master.

The user can print the report for a particular broker code or for all brokers on Pension Master. The report enables the user to:

  • find out which renewals are not yet completed.
  • find out where data is still outstanding for each renewal.
  • find out where no data has been received.

Formula File Maintenance

Revised Functions

Extended the functions avg() and avgmax() with extra arguments to allow more conditions in the selection of appropriate amounts. The User Manual pages are shown below: 

avg(field,n,fld1,fld2) OVERVIEW Use the avg function to calculate
the average of a history field over a particular number of years. SYNTAX avg(field,n,fld1,fld2) EXPLANATION field      
is a field on the salary history or contribution                
history files (pe_defsal or pe_defhst_b respectively). n              
is the number of years to average the history field over. fld1           
is the effective start conditions of the averaging. fld2           
is the basis of the averaging. NOTES
  1. If n = 0, the function will return a value of 0.
  2. n can take any value less than 40. i.e. you can average the history field over no more than 40 years.
  3. fld1 can take the alphabetic values 'e' and 'l'. 'e' will commence the averaging period from Calculation Date. 'l' will commence the averaging period from Calculation Date but will ignore the latest entry.
  4. fld2 can take the values 1, 2, 3 and 4. 1 will average extracted amounts using the highest amount if there is more than one in the same calendar year. 2 will average extracted amounts using the first amount if there is more than one in the same calendar year. 3 will average extracted amounts recorded at the renewal dd/mm as recorded on Scheme File Maintenance. 4 will average extracted amounts using the period the amount was effective for.

EXAMPLES

avg(pds_pensal,3,e,1) avg(pds_grsal,1,l,4) avg(pdb_ee_ord_c,4,e,2) 
avgmax(fld,n1,n2,fld1,fld2) OVERVIEW Use the avgmax function to
calculate the best band of consecutive years history over a given period. e.g. to
calculate the best three year average gross salary over the last ten years. SYNTAX avgmax(field,n1,n2,fld1,fld2) EXPLANATION field      
is a field on the salary history or contribution                
history files (pe_defsal or pe_defhst_b respectively). n1             
is the number of years to find the best consecutive n2                
years average over. n2             
is the number of consecutive years to average the                
history field over. fld1           
is the effective start conditions of the averaging. fld2           
is the basis of the averaging. NOTES
  1. If n1 = 0 or n2 = 0, the function will return a value of 0.
  2. n1 can take any value less than 40. i.e. you can average the history field over no more than 40 years.
  3. If n2 > number of years history for a particular member then the function performs the average over the number of years history data available. e.g. if you try to use avgmax(pds_grsal,10,3,e,1) and there is only two years gross salary details for a particular member, the function will look at consecutive two year averages of gross salary to arrive at the result.
  4. fld1 can take the alphabetic values 'e' and 'l'. 'e' will commence the averaging period from Calculation Date. 'l' will commence the averaging period from Calculation Date but will ignore the latest entry.
  5. fld2 can take the values 1, 2, 3 and 4. 1 will average extracted amounts using the highest amount if there is more than one in the same calendar year. 2 will average extracted amounts using the first amount if there is more than one in the same calendar year. 3 will average extracted amounts recorded at the renewal dd/mm as recorded on Scheme File Maintenance. 4 will average extracted amounts using the period the amount was effective for.

EXAMPLES

avgmax(pds_pensal,10,3,e,1) avgmax(pds_grsal,3,1,l,4) avgmax(pdb_ee_ord_c,4,1,e,2)

Batch Quotation Run

This facility has been extended to allow Quotation results to be finalised on a bulk basis. An extra menu item Finalise has been added to facilitate this.

After selecting the Finalise option and entering the Quotation details a screen displaying the extracted members and a finalised flag will appear. The user will be able to amend the default finalised status and also be able to deselect members from being finalised by changing their flag from 'Y' to 'N'.

If the finalised flag is shown as 'R' then this indicates that only the quotation results will be written back to the members relevant User Defined Screen. The members status, date left and the quotation finalised flag will not be updated.

If there is a problem with finalising a members quotation i.e. the quotation is not selected as being able to finalise, the quotation has not been run at the entered Calculation Date or their Finalised Status does not agree with the Finalised Status Type for the quotation, then the finalised flag will be set to 'X'. This will result in the members quotation not being finalised.

Once the extracted list has been accepted, Pension Master will finalise the quotation for those members with a 'Y' flag and write back the quotation results for those with a 'R' flag. A Finalised Report is then available which will list those members whose quotation was finalised 'Y', those whose results were written back, 'R, those who were flagged not to be finalised, 'N', and those who could be finalised with a reason why, 'X'.

Lock File Maintenance

Introduced a new Lock File Maintenance program to deal with cases where the user is trying to update information that another user is already updating. As soon as a user chooses Update in any part of Pension Master, a lock is created in the lock file which indicates what program the user is in and what item of data they are currently updating e.g. if the user is updating the contribution history of member number 13426, a lock file with Lock ID of pem_mcon13426 is created. When the user exits input mode and returns to the program menu, the relevant lock file is removed, allowing other users immediate access to that record.

If a program exits abnormally whilst the user is updating a piece of information, the lock will remain in force and the System Administrator should use the Lock File Maintenance program to remove the lock, thus allowing the user update access to the record again. Further details can be found in the User Manual.

Renewal Listing Report

This is a new report.

It can be run across all schemes on Pension Master and will show details of when the renewal date is for each scheme. The user can select the month to run the report for, if they require, for example, details of the renewals coming up next month, or they can run the report for all months. This will then show renewal dates for all schemes, grouped by month.
