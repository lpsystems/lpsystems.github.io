Release Notes 5.0.5

Details Of The Changes

  1. Date Contracted Out in the Basic Member Details screen of Member File Maintenance now defaults to Date Started Pensionable Service (pm_dps), or 6/4/1978 if later, rather than to Date Entered Scheme (pm_des), or 6/4/1978 if later, so as to allow for schemes where pensionable service commenced before the member joined the current scheme, for example in the case of a bulk buyout into a new scheme.
  2. There are occasions where members can be in a contracted out scheme but remain contracted in. To allow for this senario users will be able to update Date Contracted Out in the Basic Member Details screen to be blank but they will be prompted to confirm that they wish to leave this date blank. The same facility is available if this field is shown as Qualifying Service Date (Irish schemes only).

  3. Added a new function called exptot() which has been copied from the existing conttot() function except it calculates total expenses between two given dates for a particular expense source and investment fund. The manual page for this is as follows:

    exptot(flag,nn,date1,date2)

    OVERVIEW

    Use the exptot function to calculate the total expenses between two given dates for a particular contribution source.

    SYNTAX

    exptot(flag,nn,date1,date2)

    EXPLANATION

    flag - This is used to denote the contribution source and can take the values 1 - 6 as follows:

    1. EE Ordinary Expenses
    2. EE AVC Expenses
    3. ER Ordinary Expenses
    4. EE Protected Rights Expenses
    5. ER Protected Rights Expenses
    6. Total Expenses nn is the investment fund number you require expense totals for. Valid values are between 00 and 20. date1 is the start date for the period required. date2 is the end date for the period required.

    NOTES

    1. If date1 > date2 the value the total will be calculated with a term of 0.
    2. The two dates must be field references to date fields on the member file (pe_mem) or constant dates entered in the format YYYYMMDD where YYYY is the year, MM is the month and DD is the day.
    3. The investment fund number must be in the format nn i.e. for fund 1 you must enter 01, for fund 20 you must enter 20.

    EXAMPLES

    exptot(1,03,19950112,pm_npd)
    exptot(3,01,pm_dps,pm_dtlft)
    exptot(2,19,19950112,20031115)

  4. Amended the Actuarial Analysis Report so that it now allows you to access Quotation Results fields held on the table pe_mquote. This has necessitated the use of arguments in certain field names in a similar way to that recently introduced in the Member Report, Full Members Listing and Mail-Merge Integration Report. The fields affected are as follows:

    e.g. if you wished to output the total employee ordinary contributions for investment fund 2, you would enter pdh_tee_ordc(2).

  5. Added 'UK/Irish Scheme' indicator on Scheme File Maintenance to allow the masking of irrelevant fields to the UK or Irish market. This will also be required for the calculation of State Pension Date.
  6. Added a State Pension Date to the Member Basic Record which will automatically calculate a members SPD depending on their Date of Birth and Sex. This calculation will be carried out to all existing members on installation on this version of Pension Master and takes into account the change of State Pension Ages for women born after 6 April 1950 (UK).
  7. Added a new screen Member Indicators to Member File Maintenance. This shows whether various records exist for each member or not. There is also a Number field for Additional Benefits and Transfer Received records to show the user how many of these records are present. The records analysed by this indicator screen are as follows:
    • Part time hours
    • Benefit normination received/dependent records
    • Freehand Notes
    • Salary history
    • AVC's present
    • Additional benefits
    • Transfer in received
    • Address

    This record also holds the Ratio 1 and Ratio 2 amounts and a new field to record NI Table (UK Schemes only).

  8. Due to the addition of the State Pension Date field and the addition of the Member Indicators record, the Member Basic Details screen has been slightly reorganised. This allows for fuller descriptions of each field on-screen. Any fields that used to be on this screen and are not there now, have been moved to the new Member Indicators screen.
  9. The amendment of Company, Category and Status details on the Member Basic Record have been tightened up to try to establish a more complete history which will allow users to produce accurate Membership Movement Summaries when required. When adding a member, if the user does not enter a Deate Entered Scheme they will be warned that this date is required for the Company, Category and Status Histories and prompted to check that they wish to leave this date blank. If they do leave the date blank, they will be forced to enter an effective date of change before they will be allowed to accept the member details. When updating a member, if the Company, Category or Status fields are amended then the user will be prompted to enter an effective date of change before they will be allowed to move on to the next field.
  10. When running calculations from Bulk Quotation Run or from Member File Maintenance, the Unit Bid Price field will automatically be set to N for defined benefits schemes.
  11. Extended the selection criterea in Batch Quotation Run to allow selection using Special Codes.
  12. Amended the 'Retire at end of month' field on the Category File Maintenance to 'Normal Pension Date Adjustment'. Using the available codes you can set the categories NRD to be:
    • the exact day of the member reaching NRA (N)
    • the first day of that month (S)
    • the last day of that month (E)
    • the last day of the previous month (P)
    • the first day of the following month (F)
    • a fixed day in the month (D)

    If any of your categories are currently set up with the 'Retire at end of month' as Y, the installation program will automatically convert this to the correct new code (E).

  13. Replaced the User Defined Amounts and User Defined Dates screens with new, more flexible screens that allow you to customise them as you wish. You can now set up as many User Defined Screens and User Defined Arrays as you wish rather than being limited to one screen of values and one screen of dates as used to be in Pension Master. These screens can easily be customised as the user requires so that you can hold any conceivable item of data.
  14. User Defined Screens will allow users to set up their own screens with up to twenty fields. Each field is set up with a format of any one of the following:
    • Date (DT)
    • Indicator (IN)
    • Numeric Integer (NI)
    • Numeric Decimal (ND)
    • Header (HD)
    • Text (TX)

    This then enables the system to validate the users input onto the screen in a members record e.g. the system will not allow a date to be entered in a field set up as an indicator.
    User Defined Arrays follows the same principle except it shows a historical list of up to five columns with the same format controls as User Defined Screens. The only exception to this is that the use of Header types are not allowed in User Defined Arrays.
    These screens are displayed in the menu selection screen with the screen description and can therefore be accessed directly rather than having to select the User Defined Screen option then having to select the particular screen that you want. Only the screens and arrays that exist for the scheme of the member selected will be displayed on the menu selection list and similarly on the member report screen selection.

  15. Added new menu option under System Data called User Defined Messages. This will allow users to set up more meaningful and specific warning messages that they can use in formulae. The messages can be up to 50 characters long and are selected in formulae using the syntax 'mess(n)' where n is the number allocated to each message. An F8 lookup is available within Formula File Maintenance when you enter mess at the start of a formula line. This will display a list of the available messages and, when the relevant message has been selected, the system writes back to the formula line the required code in the correct format. * Systems Administrators will need to add this menu option to menu listings and users accounts (see attached notes).
  16. Allow formula codes to be up to 8 characters long rather than the old 3 characters. Also, check to make sure that the user does not embed spaces in the formula code.
  17. Amended the prints generated from the Member Contributions Screen to show Totals To Renewal Date less 1 day and Totals To Date.
  18. Added a new screen Additional Benefits to Member File Maintenance. This record differs from a Transfer-in as a Fixed Benefit or a Service amount the member has received from the scheme or a scheme which comes under the same scheme unmbrella e.g. service in a scheme that has merged into the current scheme, strike periods. These benefits must be entered against a Benefit Code which must be set up under Additional Benefit Codes Maintenance which is a new option in the System Data sub menu. * Systems Administrators will need to add this menu option to menu listings and users accounts (see attached notes).
  19. Amended the report print from Category File Maintenance to number the Quotation Type results on the output. This should help users if they wish to refer to previous results in formulae by displaying the result number rather than the user having to count down the list.

  23. Added new menu called QUOTES to the Quotations Screen in Member File Maintenance. This has the following options:
    • Add - add a new quotation
    • Query - search for an existing quotation
    • Run - run an existing quotation
    • Finalise - finalise an existing quotation
    • Un-Finalise - un-finalise an existing quotation
    • Override - manually update existing quotation results

    When the user first selects the Quotations Screen they are presented with a pop-up picklist of all the available quotations for the current member. Simply use the arrow keys to select the required quotation and press Return to view the details.

  24. Allow the user to write the results of finalised quotations back to the member record. This is done by setting up a User Defined Screen to hold the details you wish to write back from you quotations. e.g. you might set up a User Defined Screen called 'REFUND' to hold details of the results of refund of contributions calculations that have been finalised. In this screen you would set up each field that you wish to hold data for e.g. net refund, tax payable, state premium payable. These fields must be set up as numeric decimal (ND) fields so as they can correctly display your calculation results.
  25. Added new fields WRITE and NUM to the Benefit Definitions screen of Category File Maintenance. These are to be used in conjunction with finalising quotations (see previous section for further details). The WRITE field indicates which User Defined Screen you wish to write the results of the current formula back to when quotes are finalised for this Quotation Type e.g. you would enter 'REFUND', following on from the example in the previous section. The NUM field indicates which field on the User Defined Screen you wish to write the results of the current formulae to e.g. if you wish to write the result of the new refund of contributions back to the first field on the 'REFUND' screen, enter '1' in the NUM field.

Future Developments

The following list will detail some of the planned changes to Pension Master in the future.

  • Only display Quotation Types relevant to the members current status. This would be controlled in the Quotation Types screen in Category File Maintenance by a code that would match the members' status type e.g. A - Actives. Only members currently with this status type would have this Quotation Type displayed. A code of G would be created as a global code which would allow the Quotation Type to be available to all status types.
  • Allow write back of status and leaving date on finalising Quotation results. This would work in a similar way to the above except when the user tried to finalise the Quotation results, they would be presented with a pop up list of applicable status codes in the specified status type group.
  • Ability to cope with monetary values in an EMU currency.
  • Data Clean reports. This would be released in a basic form initially and will warn users of missing or conflicting member data.
  • Allow the deletion of individual records for a member e.g. salary details, rather than the complete record.
  • Allow historic interest and factor tables. This would enable Quotations to be run using interest rates and factors relevant to the quote date.
